programming languages

programming languages are a hobby of mine. here's a few that i made.

Wisp (2013)

my first real lisp! written in Haskell during the winter 2013 batch at Recurse Center (at the time called Hacker School). i had already messed around a little bit with Common Lisp due to StumpWM and had just been introduced to SICP. Wisp was intended to be the scripting language for a player-programmable MUD i was writing, but it ultimately grew to eclipse the rest of the project. apart from being written in Haskell and running inside a text-based multiplayer game, it was a pretty standard interpreted Lisp-1. one slightly unusual feature was automatic currying.

Shimmre (2014)

a self-hosting parser generator language written in CoffeeScript, for some reason! inspired by Ometa. you would write a grammar in Shimmre and provide transformation rules in a host language, and Shimmre would generate host language code. it had JavaScript and Ruby backends.

Gus (2015)

i wanted to write another Lisp, in C this time. Gus was a single C file Lisp interpreter named after a coworker's cute dog. it used a very standard mark-and-sweep garbage collector.

Mattie (2016)

a kind of weird experimental string-centric esolang, based on parsing expression grammars, like Shimmre. Mattie (named after my dear cat) was written in Scheme with a syntax based on Shimmre's metagrammar, but rather than generating host language code it was entirely based around transforming strings.

Lisa (2020)

aka Lips. originally meant to be an experiment in FORTH-like languages, it turned into another Lisp written in C, but this time with much cooler features: copying garbage collection, a self-hosting threaded code compiler, actual compile-time macros, and a very stripped-down syntax based on just 5 special forms. still in development!

K2 (2022)

a microKanren derivative written in Lisa that came out of my research into relational languages at Oso. being based on microKanren it's obviously extremely minimal. the main innovation is to represent result streams uniformly as Church-encoded lists, which removes a layer of runtime reflection; on the other hand it creates a lot more closures, so the performance picture is complicated. even so i think it's a very aesthetic approach to take! it would be much less tolerable in Scheme, but Lisa's lambda syntax is so clean it's basically transparent.
